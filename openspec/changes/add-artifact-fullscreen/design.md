# Artifact full-screen design

<!-- cspell:ignore fullscreenchange -->

## Context

Owner Preview and public Viewer routes currently stream the Artifact entry file as the top-level document. That preserves relative paths but leaves no trusted ShareSlices surface in which to place a full-screen control. Artifact grid thumbnails also link directly to the raw authenticated Preview route, so a card cannot enter full screen and then return to unchanged management state.

Artifact HTML is untrusted, and `PRODUCT.md` prohibits rewriting it. Version 0.0.1 nevertheless preserves accepted HTML and JavaScript behavior and explicitly defers stronger Viewer isolation. The full-screen design must therefore add trusted controls without presenting the current same-Origin deployment profiles as isolated.

The browser Fullscreen API requires transient user activation for `requestFullscreen()`, can reject the request, and can end a session independently. Nested content requires explicit Fullscreen Permissions Policy. The relevant browser contracts are the [Fullscreen API Standard](https://fullscreen.spec.whatwg.org/) and the [HTML iframe rules](https://html.spec.whatwg.org/multipage/iframe-embed-object.html).

This change realizes a small Web content-player surface plus a Viewer HTTP-shell extension within the current Web and Hono runtime seams in `docs/design/modules.md`. It does not add an application-domain service or another runtime.

## Goals / Non-Goals

**Goals:**

- Let an Owner enter Full-screen Preview directly from an eligible Artifact grid card and return to unchanged management state.
- Let an Owner or Viewer enter and exit full screen from a trusted control around ordinary Preview or accessible Viewer content.
- Keep the effective Version, authorization, Publication, Share link, relative references, and no-cache behavior unchanged.
- Preserve Artifact content's existing user-activated full-screen capability.
- Keep failure and browser-driven exit behavior explicit, observable, and testable.

**Non-Goals:**

- Introduce a Play domain concept, presentation lifecycle, new Version selection, or Publication mutation.
- Add controls to list rows, selection mode, or non-content Viewer status pages.
- Rewrite Artifact HTML, support root-absolute Artifact URLs, or introduce mobile and tablet behavior.
- Deliver the future separate-site isolation, strict CSP, Permissions Policy hardening, or separate Preview grant described by the existing security roadmap.
- Add a database migration, Worker operation, CLI behavior, dependency, or ADR.

## Decisions

### 1. Put untrusted content inside a trusted player shell

Owner Preview uses a route-level `ArtifactPreviewPage` under the authenticated `/artifacts` Web surface. The page renders a reusable `ArtifactPlayer` whose full-size iframe loads the existing Cookie-authenticated `/api/versions/{versionId}/content/` entry route. Grid card Preview links and the Artifact detail Preview action open this trusted page in a new tab with no opener; raw content remains an internal frame target rather than the normal Preview destination.

Public `/a/{shareSlug}/` keeps the stable Share-link address but returns a minimal trusted Viewer player document when the Publication is accessible. Its iframe loads the same route with a reserved content-mode query parameter. That parameter makes the initial iframe entry request stream the manifest entry file; subsequent root/self navigation in that iframe is recognized from the browser's `Sec-Fetch-Dest: iframe` request context so dropping the query cannot recursively load the player. Existing `/a/{shareSlug}/{assetPath}` resolution remains unchanged. Because a query does not change the URL path base, `assets/app.js` still resolves to `/a/{shareSlug}/assets/app.js`. The parameter and request context are routing state, not authorization: every entry and asset request continues to resolve the current Publication independently.

The Viewer player document is generated by the Viewer HTTP adapter and contains no Artifact or Owner metadata. It uses fixed trusted markup, styles, Lucide-equivalent maximize/minimize SVG paths, and a small fixed controller script. This avoids loading management code on the configured Viewer Origin and works for Compose, Kubernetes addresses without DNS, and public Viewer domains. The OpenAPI entry response and focused contract tests are updated to distinguish player and content-mode responses.

Alternatives rejected:

- Injecting controls into the Artifact entry HTML violates the no-rewrite product contract and risks corrupting arbitrary documents.
- Making user HTML implement the control cannot provide consistent behavior or trusted exit UI.
- Moving Artifact content under a new path prefix changes the relative URL base and can collide with Artifact paths.
- Opening a new window and asking that window to enter full screen cannot reliably retain the originating click's transient user activation.

### 2. Request full screen synchronously from the clicked Card

Each eligible `ArtifactTile` keeps a ref to its existing card player target. The bottom-right maximize button stops card-link propagation, marks the latest ready Version active, and invokes `requestFullscreen()` on that already-connected target before the user activation expires. The card's full-screen presentation replaces its normal thumbnail/footer rendering with `ArtifactPlayer`; no route navigation or global management-state reset occurs.

On request rejection, the card clears its temporary player state and reports a Sonner error. On `fullscreenchange`, losing that card as the full-screen element clears the player state. This covers the exit icon, native Escape behavior, and browser-initiated termination without intercepting Escape itself. Search, filters, view mode, selection, and scroll therefore remain owned by the still-mounted `ArtifactsPage`.

The control appears persistently beside the existing bottom-right Publication action only when the card has a latest ready Version and `preview` is allowed. It is absent in selection mode and list view. The thumbnail's existing ordinary Preview navigation remains independent.

Alternative rejected: mounting a new portal only after the click risks calling `requestFullscreen()` after React commits and the transient activation has expired.

### 3. Use one owner-side player contract for Card and Preview page

`ArtifactPlayer` owns the iframe, dark neutral full-viewport background, top-right shadcn Base UI icon button, Lucide maximize/minimize icon, tooltip, accessible name, inline failure state, and the focused controller logic for `requestFullscreen()`, `document.exitFullscreen()`, `fullscreenchange`, cleanup, and normalized failure reporting. Card mode supplies the card target and closes on exit; Preview-page mode supplies the page player target and stays mounted on exit.

The content viewport fills `100vw` by `100vh` in full screen. ShareSlices does not scale or impose an aspect ratio on the Artifact; its own layout receives the available iframe viewport.

The public Viewer shell implements the same observable contract without importing Web internals across the Web/API boundary.

### 4. Preserve current Artifact capabilities without claiming sandbox isolation

Player iframes explicitly allow Fullscreen Permissions Policy so Artifact content such as an embedded video can retain user-activated full-screen behavior. The parent player also listens to full-screen state changes and offers an exit action whenever its document has a full-screen element.

The iframe does not add a `sandbox` attribute in version 0.0.1. A restrictive sandbox would change accepted Artifact JavaScript, navigation, storage, or fetch behavior, while combining `allow-scripts` and `allow-same-origin` for same-Origin content is not an isolation boundary: the HTML Standard notes that such content can remove the sandbox and reload. Public production already assigns the Viewer a separate configured Origin, but local, Compose, shared-test, and intranet topology may share an Origin according to existing deployment policy. This change preserves that accepted risk and adds no security claim.

Alternative rejected: `sandbox="allow-scripts allow-same-origin"` would look protective without constraining same-Origin Artifact code. A future isolation change can move raw content to a separate content Origin and then apply a capability-minimized sandbox without coupling that migration to full-screen UX.

### 5. Treat Fullscreen API failure as a presentation failure

The button calls `requestFullscreen()` only inside a user event and never automatically on page load. Unsupported or rejected requests do not change URLs or retry. Card mode closes its temporary player and leaves management unchanged; Preview and Viewer page modes retain normal rendering and expose a concise error status. A subsequent explicit click is a new request.

The UI derives enter/exit state from `document.fullscreenElement` after `fullscreenchange`, rather than assuming a resolved request remains active. This accounts for native Escape and the browser's right to end full screen independently.

### 6. Keep server authorization and caching at every content request

The owner player iframe continues to use the existing Preview route, so every entry and asset request validates the current management session, ownership, ready-Version state, and manifest path. The Viewer shell and its content-mode request independently resolve the Share slug and current accessible Publication; a later Expired or Unpublished request produces the existing non-content state instead of content.

All shell, entry, asset, and status responses remain `Cache-Control: no-store`. Object storage remains private, and neither shell receives an object-storage URL. Full-screen state is browser-only component state and is not persisted or sent to the API.

## Risks / Trade-offs

- **Same-Origin Artifact code can influence its parent player shell in deployment profiles that share an Origin** → Preserve the explicitly accepted 0.0.1 risk, keep management operations server-authorized, expose no new credentials or data to the shell, and leave real content-Origin isolation to the existing hardening roadmap.
- **The Viewer content-mode query is visible to Artifact JavaScript through `location.search`** → Reserve one fixed parameter, keep the public Share link free of it, document it in the HTTP contract, and prefer this limited difference over HTML rewriting or changing the relative path base.
- **Browser full-screen behavior differs or can be denied by policy** → Treat request failure as recoverable UI state, derive state from browser events, and keep normal content usable.
- **An Artifact's own nested full-screen request can replace the shell target** → Allow it deliberately for compatibility and track the nested state from the parent document. Browsers place the full-screen iframe above its trusted siblings, so this nested case exits through Escape or another browser-provided exit; the trusted control resumes its synchronized state after that exit.
- **The public Viewer shell duplicates a small amount of owner-side controller behavior** → Keep its fixed script minimal; do not couple Web and API build artifacts or introduce a shared cross-runtime UI package for one control.
- **Changing `/a/{shareSlug}/` from raw Artifact HTML to a shell changes response-body expectations** → Update OpenAPI and route contract tests while keeping URL, status, no-cache headers, Publication resolution, and rendered content behavior stable.

## Migration Plan

1. Land the Web owner player, Preview page route, card entry, and focused Fullscreen API tests.
2. Land the Viewer player response and content-mode routing with OpenAPI and HTTP contract coverage in the same change.
3. Run the Web, API, and full repository quality gates, then exercise Card Preview, ordinary Preview, Published Viewer, Expired, Unpublished, relative assets, user-content full screen, Escape, and rejected requests in a supported desktop browser.
4. Deploy Web and API artifacts together. No database or object migration is required.
5. Roll back both artifacts together if needed; existing Artifact, Version, Publication, Share-link, and stored-content data remains compatible.

## Open Questions

None. Product entry points, terminology, exit behavior, Version selection, control placement, content capability, failure behavior, and desktop acceptance were resolved during the design session.
