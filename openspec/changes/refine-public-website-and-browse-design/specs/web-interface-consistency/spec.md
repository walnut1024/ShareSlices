# web-interface-consistency Delta Specification

## ADDED Requirements

### Requirement: Present one coherent public Website frame

The Web SHALL render the Website homepage, Browse, trusted Gallery listing, and public Creator surfaces with one public visual language based on the existing Geist typography, semantic color tokens, shadcn/Base UI controls, compact 64-pixel navigation, restrained borders and shadows, and a consistent content frame no wider than 1200 pixels. The public header SHALL expose only route-owned Website, Browse, Session, and Console destinations applicable to the current state. A public footer SHALL contain only implemented route-owned destinations and MUST NOT advertise placeholder Company, Legal, Status, Changelog, API, category, or Creator-directory addresses.

At every supported desktop viewport, the public frame, primary content, search, collection controls, cards, account controls, and footer MUST remain readable and reachable without unintended clipping or horizontal overflow. Active navigation, focus, unavailable content, and other state MUST remain distinguishable without relying only on color.

#### Scenario: Signed-out visitor opens a public page

- **WHEN** a signed-out visitor opens `/` or `/browse` at a supported desktop viewport
- **THEN** the Web renders the same compact public header, content frame, typography, surface treatment, and real route destinations on both pages without clipping or horizontal overflow

#### Scenario: Signed-in User opens a public page

- **WHEN** a signed-in User opens `/` or `/browse`
- **THEN** the public frame remains unchanged except for the existing Session-aware Console and account controls, whose reserved space does not shift the primary navigation

#### Scenario: Visitor reaches the public page ending

- **WHEN** a visitor reaches the end of a public Website page
- **THEN** the Web presents a bounded public footer containing only implemented destinations generated by the route model

#### Scenario: Visitor uses keyboard navigation

- **WHEN** a visitor tabs through public navigation, search, collection controls, cards, tags, pagination, and footer links
- **THEN** each interactive element exposes a visible focus treatment, a meaningful accessible name, and a keyboard order that follows the visible page order

### Requirement: Make the Website homepage search-led and content-led

The Website homepage SHALL begin with a centered light-surface introduction containing a plain product promise, the real Gallery search, a Browse destination, and the existing Session-aware publishing destination. Real Gallery discovery SHALL remain the primary visual evidence of the product and SHALL use the existing Featured result with its conditional Newest fallback. The page SHALL retain a compact explanation of Upload, Publish, and Share safely and MAY repeat the real publishing and Browse destinations near the page ending.

The homepage MUST NOT use a decorative mock browser or fabricated interface as its dominant product evidence and MUST NOT present fictional aggregate counts, public play counts, engagement-ranked collections, category taxonomy, an independent Creator directory, **Open app**, or unsupported destination links. Loading, empty, and Gallery-unavailable results SHALL occupy a bounded discovery region that preserves the page hierarchy and does not leave an oversized blank area.

#### Scenario: Homepage discovery returns Artifacts

- **WHEN** the homepage resolves one or more eligible Gallery cards
- **THEN** the Web presents the search-led introduction followed by the real Gallery cards, a route-owned Browse destination, the publishing explanation, and no unsupported prototype claim or destination

#### Scenario: Homepage discovery is loading

- **WHEN** the homepage Gallery request has not resolved
- **THEN** the discovery region presents a labelled skeleton layout with approximately the same bounds as the resulting Gallery grid while the introduction and publishing journey remain usable

#### Scenario: Homepage discovery is empty

- **WHEN** the homepage Featured request and its permitted Newest fallback resolve with no eligible Artifacts
- **THEN** the discovery region explains that no public Artifacts are available without collapsing the page structure or presenting fictional examples

#### Scenario: Homepage Gallery is unavailable

- **WHEN** Gallery availability resolves unavailable on the homepage
- **THEN** the Website retains its product introduction and publishing journey, replaces discovery with a bounded unavailable explanation, and removes active Gallery search and Browse actions without exposing Gallery resource evidence

### Requirement: Present Browse as a bounded discovery workspace

`/browse` SHALL render inside the same public frame with a Home/Browse breadcrumb, one clear page title and supporting description, a labelled Gallery search, and one grouped collection control containing only the implemented default, Featured, and Newest choices. Search, exact-tag, collection, empty, unavailable, and cursor-pagination states SHALL retain that page context and reflect the route-owned query state.

The results region SHALL use a dense four-column Gallery-card grid at supported desktop widths. Its controls and cards MUST fit within the 1200-pixel frame at the 1280-pixel support boundary. Browse MUST NOT derive or display a category sidebar, popular-tag ranking, aggregate result counts, Trending, Most played, or another unsupported filter from the current result page.

#### Scenario: Visitor opens default Browse

- **WHEN** a visitor opens `/browse` with eligible Gallery results
- **THEN** the breadcrumb, page header, search, default collection choice, four-column card grid, and available pagination form one continuous discovery workspace

#### Scenario: Visitor opens a collection or query result

- **WHEN** a visitor opens Featured, Newest, search, or exact-tag Browse state
- **THEN** the page keeps the same header and control geometry, identifies the active route-owned state, and renders only the eligible cards returned for that state

#### Scenario: Browse returns no matching results

- **WHEN** a Browse collection or query resolves with no cards
- **THEN** the results region presents a bounded empty explanation tied to the current state without removing the search and supported collection controls

#### Scenario: Visitor loads another page

- **WHEN** a Browse result includes a cursor and the visitor activates **Load more**
- **THEN** the pending state remains perceivable, the existing cards stay stable, and the appended cards retain the same grid and keyboard order without a duplicate request

#### Scenario: Browse renders at the supported-width boundary

- **WHEN** `/browse` renders at a 1280 by 720 viewport with search and collection controls visible
- **THEN** the header, controls, four-column grid, and account action remain inside the viewport without unintended horizontal scrolling or hidden primary actions

### Requirement: Give Gallery cards a stable trusted interaction hierarchy

Gallery cards on the homepage and Browse SHALL use the same cover ratio, neutral placeholder treatment, compact metadata hierarchy, bounded text, tag treatment, and border, radius, hover, and focus language. The trusted listing destination SHALL remain visually primary and SHALL NOT imply that the Artifact executes within the card. Exact-tag actions SHALL remain separate valid links outside the listing link, and the card MUST NOT nest interactive elements.

The card presentation MUST NOT invent a play action, play count, editorial badge, or another field absent from the Gallery projection. A missing cover, description, or tag set SHALL preserve card geometry and the remaining metadata rather than leaving a broken or misleading control.

#### Scenario: Card has a ready cover and complete metadata

- **WHEN** an eligible Gallery card has a ready static cover, description, creation time, and tags
- **THEN** the Web presents the trusted listing destination as the primary card interaction and presents each tag as a separate exact-tag Browse action

#### Scenario: Card cover is unavailable

- **WHEN** a Gallery card cover is pending, absent, or terminally failed
- **THEN** the card presents the neutral placeholder with the same metadata hierarchy, dimensions, listing destination, and focus behavior as a ready-cover card

#### Scenario: Card has optional metadata missing

- **WHEN** a Gallery card has no description or no tags
- **THEN** the card preserves a coherent compact layout without placeholder copy, empty interactive controls, or invented metadata

#### Scenario: Visitor moves between card interactions

- **WHEN** a visitor uses the keyboard to move from a card's listing destination to its tag actions
- **THEN** focus follows the visible order, each destination is individually named and visible, and no nested-interactive or duplicate-navigation behavior occurs
